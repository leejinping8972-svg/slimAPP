# Design Spec — luckdate 业务文档包

> 日期：2026-07-24  
> 状态：已口头批准（目录 A1 + 纯业务边界 + 品牌名 luckdate）  
> 产物：PRD + 规则补充分册（不含技术落地）

## 目标

把分散的 PRD、交互说明、Sunny 对话规则、评分规则、管理后台说明、用户补充闭环规则，整理成一套 **纯功能交互 + 业务需求** 文档包，供产品/设计/运营/研发对齐「做什么」，不规定「怎么实现」。

## 品牌

- App 正式名称：**luckdate**
- AI 助手名：Sunny
- 历史名 ChatViva Slim 仅作背景说明，不作为产品对外主名

## 写作边界

**写入：** 用户路径、页面职责、业务规则、异常与兜底、成功终态、运营后台业务能力。  
**不写：** 接口、Schema、客户端/服务端架构、模型供应商、数据库、代码路径。

## 文档包（仓库根目录）

| 文件 | 职责 |
|------|------|
| `文档索引.md` | 阅读顺序与权威优先级 |
| `PRD-luckdate.md` | 主 PRD（8 段模板） |
| `规则-Sunny对话与意图路由.md` | Sunny 能办什么事、安全与优先级 |
| `规则-每日记录与评分.md` | 记录项与评分业务口径 |
| `规则-管理后台.md` | 运营/客服后台业务能力 |
| `规则-开通复购与优惠券.md` | 关联、内购确认收货、券、Day28 A/B/C |

## 权威顺序

冲突时：`PRD-luckdate.md`（做什么）→ 各 `规则-*`（怎么判定）→ 旧交互/变更稿（历史参考）。
